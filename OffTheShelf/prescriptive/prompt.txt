System Identification Model: Iterative Training & Validation

Objective

You are a coding agent tasked with building, training, and iteratively refining a System Identification (Sys ID) model using PyTorch. You will work with time-series data to learn a dynamical system's behavior via full-day autoregressive rollouts (defined precisely below). Your work should read as a clear, well-reasoned engineering narrative — at every stage we want to understand what you observed, what you changed, and why you changed it. The story of how the model evolves matters just as much as the final result.

- Development data: ./dev/dev_300.csv — Dev data at the same resolution. Use this as a held-out set to monitor generalization. Do not use it to make training decisions such as early stopping or hyperparameter selection during the iterative refinement loop.
- Model interface contract: ./load_model.py — This script defines the exact interface your model must conform to. Read it carefully. Your trained model must be loadable by this script, which means you must produce:
  - A Python module called hvac_sysid_model.py that defines the HVACSysIdModel class, plus normalize_array and denormalize_array helper functions, exactly as load_model.py imports them.
  - A .pt checkpoint file containing the keys that load_model.py expects: model_state_dict, classification, norm_stats, timestep_seconds, context_length, hidden_size, num_layers.

Each seed CSV has 288 rows (one full day at 5-minute resolution), with zone temperatures present for the first 12 rows (1 hour of context) and NaN thereafter.

Key Definition — Autoregressive Rollout

During evaluation (and optionally during training), the model operates autoregressively for an entire day:

1. The model is seeded with up to W true timesteps from the beginning of the day, where W is the model's chosen input window length.
2. The model predicts the next single timestep.
3. That prediction is appended to the history, the window slides forward by one step, and the model predicts the next timestep using the updated window — which now contains its own prior predictions.
4. This continues until the entire day has been predicted.

Input Window Constraint

The model may use 1 to 12 timesteps (up to 1 hour at 5-minute resolution) as its input window to predict the next timestep. This is a hard upper bound. Using a shorter window is perfectly acceptable and encouraged if it performs well — a leaner model that trains faster is preferred over a marginally better model that is slow.

Step 2 — Iterative Refinement Loop (Iteration 2, 3, …)
Repeat the following until the convergence criterion is met:

1. Diagnose the current model's performance. Before changing anything, write a short analysis paragraph addressing questions such as:
   - Is the model overfitting (training loss low but rollout RMSE high)?
   - Is the model underfitting (high training loss, poor expressivity)?
   - Is it merely tracking the mean and not capturing dynamics?
   - Are errors accumulating rapidly during rollout (early timesteps accurate, late timesteps diverge)?
   - Are certain days significantly worse than others — and if so, why might that be?
   This diagnosis is the most important part of each iteration. Be specific and reference actual numbers.

2. Propose changes. Based on your diagnosis, state clearly what you intend to change and why that change addresses the diagnosed issue. Changes may include (but are not limited to): architecture depth/width, activation functions, input window length (within the 12-timestep cap), learning rate, number of training epochs, regularization, dropout, teacher-forcing schedule, loss function, data augmentation, or normalization strategy.
   Balance complexity and training time. Prefer simpler, faster-to-train changes first. Do not jump to a massive model if a learning rate adjustment might suffice. If a change significantly increases training time, acknowledge the trade-off explicitly.

3. Print a full Model Identity Card for the revised model (even if the architecture didn't change — hyperparameters and training config matter too).

4. Train the revised model from scratch (or fine-tune — state which and why).

5. Evaluate via full-day autoregressive rollouts on the training set, computing per-day RMSE and overall training RMSE exactly as in Step 1.

6. Convergence check:

   After each iteration, evaluate the model's quality using the criteria below. Convergence requires meeting BOTH Condition A (absolute quality) AND Condition B (diminishing returns).

   **Condition A — Absolute RMSE Quality Gate (Temperature Predictions)**
   The data is at 5-minute resolution (288 steps/day, 12 steps/hour). The target RMSE threshold is ≤ 0.40°C.

   The overall training RMSE (across all days) must be at or below this threshold.

   **Condition B — Diminishing Returns Check**
   The improvement in overall training RMSE between the current iteration and the previous iteration must be less than 0.02°C (0.036°F):

       |RMSE_prev − RMSE_current| < 0.02°C

   This indicates the optimization has effectively plateaued and further architectural or hyperparameter changes are unlikely to yield meaningful gains.

   **Convergence Logic:**
	if (overall_RMSE ≤ 0.40) AND (|RMSE_prev - RMSE_current| < 0.02):
	→ CONVERGED. Proceed to Step 3.
	elif (overall_RMSE ≤ 0.40) AND (improvement ≥ 0.02):
	→ Quality target met but model is still improving. Run ONE more iteration
	to confirm stability, then converge if the quality gate still holds.
	elif (overall_RMSE > 0.40) AND (|RMSE_prev - RMSE_current| < 0.02):
	→ Plateaued above target. The model has structural or data limitations.
	Print a warning noting the gap between current RMSE and target.
	Attempt at least one more iteration with a meaningfully different
	approach (e.g., architecture change, different window length,
	normalization strategy). If the next iteration still does not break
	through, proceed to the guard-rail.
	else:
	→ Model is still improving and has not yet reached quality target.
	Continue iterating.

**Supplementary Diagnostics (print at every iteration):**
In addition to overall RMSE, compute and print the following at each iteration to guide diagnosis:
- **Mean Bias Error (MBE):** Average signed error across all rollout timesteps. Should be near zero; a persistent nonzero MBE indicates systematic drift. Flag if |MBE| > 0.1°C.
- **Max Absolute Error:** The single worst prediction error across all days. Flag if this exceeds 2.0°C — it may indicate a day with unusual disturbances or a regime the model cannot represent.
- **Worst-Day RMSE vs. Best-Day RMSE:** Print both values and their ratio. A ratio above 3:1 suggests the model struggles with specific operating conditions and may benefit from targeted investigation of those days.
- **Early vs. Late Rollout Error:** Split each day's rollout into the first half and second half. Compute RMSE for each half separately and report the average across days. If late-half RMSE is more than 2× early-half RMSE, error accumulation is a dominant issue — consider strategies like scheduled sampling / curriculum teacher forcing, shorter input windows to reduce compounding, or training directly on multi-step loss.

**Guard-rail:** If you reach 5 iterations without convergence, stop the loop, select the iteration with the lowest overall training RMSE as the best model, print a warning that includes:
- The final RMSE achieved vs. the target threshold (0.40°C).
- A brief hypothesis for why the target was not met (e.g., data quality, missing features, insufficient model capacity, irreducible noise floor).
- Then proceed to Step 3.
